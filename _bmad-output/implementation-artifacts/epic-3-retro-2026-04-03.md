## Part 1 — Epic review

### What went well

1. **Durable foundation first:** `3-1` landed `issues` + `issue_change_events` with append-only history and org-scoped queries before HTTP and UI work piled on — later stories could focus on behavior, not schema churn.
2. **Vertical slices:** Create → list → detail → patch → picker → timeline → filters → a11y kept each story shippable and reviewable without a big-bang merge.
3. **RBAC reuse:** `TenantPathAuthorization`, `RoleAuthorization`, and `docs/rbac-matrix.md` carried straight into issue routes — no parallel permission system.
4. **Review culture:** Story files capture **Review Findings** with patches applied (OpenAPI clarity, invalid query ITs, UX-DR6 Retry on PATCH failures, normalization vs `wouldChange`, sprint status timestamp hygiene).
5. **Explicit MVP boundaries:** Status transition matrix and CLOSED immutability policy documented in `3-5`; concurrent conflict handling explicitly deferred where product allows last-write-wins.

### Challenges and patterns (from Dev Notes + review threads + `deferred-work.md`)

| Theme | Occurrence | Systems takeaway |
|--------|------------|------------------|
| **`IssueStubController` naming** | 3.2, 3.3, 3.5 | Real routes still live under a “stub” class name — rename or split when the next refactor touches the controller surface (optional cleanup, not blocking). |
| **OpenAPI vs implementation** | 3.5, 3.8 | Reviews repeatedly tightened `@Operation` / `@Schema` / error docs — keep Springdoc DTOs as first-class when adding endpoints. |
| **Mutation UX parity** | 3.5 | Detail load had Retry; PATCH errors needed the same — **UX-DR6** is easier to miss on mutations than on initial fetch. |
| **Empty-state complexity** | 3.3, 3.8 | Probe queries and multi-dimensional list keys are correct but easy to regress — document query-key dimensions in one place when touch points multiply. |
| **History paging** | 3.7 (deferred) | First page only (`size=50`) — acceptable MVP; product must signal when “load more” becomes required. |
| **Theme vs contrast gate** | 3.9 (deferred) | Node contrast script duplicates hex from `theme.ts` — single source of truth when theme churn increases. |
| **Optimistic locking** | 3.1 (deferred) | Overlapping edits remain last-write-wins — acceptable until conflict UX is a requirement. |

### Deep story synthesis — recurring technical themes

- **Integration-first API proof:** `IssueListIT`, `IssueStubController` routes, RBAC negatives, and validation errors stayed in step with feature work.
- **Mobile:** Stable `queryKey` shapes, `invalidateQueries` after mutations, and EmptyState variants for “no issues” vs “filters hid everything” show up across multiple stories — good pattern to preserve for Epic 4 notification lists.
- **Technical debt is visible:** `_bmad-output/implementation-artifacts/deferred-work.md` captures cross-story deferrals with story attribution — reduces surprise for Epic 4.

## Part 2 — Next epic preparation (Epic 4)

**Epic 4 title (from `epics.md`):** Awareness — in-app notifications & push.

**Goal:** Meaningful issue events produce notifications, delivered in-app and via push per role/rules — awareness without replacing the issue aggregate as source of truth.

**Dependencies on Epic 3 (readiness):**

- **Issue mutations and history** — **ready** as event sources; `IssueChangeType` and service-level updates give hooks for taxonomy mapping (`issue.created`, `issue.assigned`, `issue.status_changed`, etc., per architecture).
- **Issue list/detail** — **ready** for deep links and notification rows that reference `issueId`.
- **RBAC + org scope** — **ready** for fan-out rules by role; Epic 4.2 will encode the matrix.
- **Mobile shell + tabs** — **ready** for Notifications tab (UX-DR7, DR11) once REST exists.

**Gaps to watch:**

- **Persistence:** Notification tables and outbox-style rows are **Epic 4.1** scope — not assumed in Epic 3 schema.
- **Domain events vs DB history:** Align emitted notification taxonomy with existing `IssueChangeEvent` semantics to avoid duplicate or conflicting meanings.
- **Deferred global items:** HTTP client timeouts, OpenAPI CI smoke, and `IssueStubController` rename are still cross-cutting — decide whether any belong in the first Epic 4 story for “clean runway.”

### Significant discovery assessment

**Epic update required before Epic 4:** **No** — Epic 3 delivery matches the planned FR11–FR19 path; Epic 4 stories in `epics.md` remain valid. Normal refinement per story (e.g. first notification record shape, idempotency) is expected.
