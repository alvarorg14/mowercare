## Part 1 — Epic review

### What went well

1. **Single enforcement spine:** `RoleAuthorization` + `TenantPathAuthorization` + stable `code` values (`FORBIDDEN_ROLE`, `TENANT_ACCESS_DENIED`, conflicts, last-admin codes) kept API behavior consistent from stubs through real user routes.
2. **`docs/rbac-matrix.md` as living contract:** Updated across 2.1–2.6; mobile and API trace to the same matrix.
3. **Story sequencing:** 2.1 stubs → 2.2–2.4 data and auth paths → 2.5 UI → 2.6 product guardrails avoided building a full admin UI before APIs were trustworthy.
4. **Concrete review fixes:** Examples: duplicate-email mapping via Hibernate constraint names; `ON DELETE SET NULL` on `deactivated_by` FK; injected `ObjectMapper` in `AccountStatusVerificationFilter` for Problem Details parity; mobile copy gated by role on Settings.
5. **Explicit edge rules:** Last-admin demotion/deactivation, idempotent PATCH/deactivate, JWT role lag called out for mobile — reduces surprise in Epic 3 when permissions matter on issue routes.

### Challenges and patterns (from Dev Notes + review threads)

| Theme | Occurrence | Systems takeaway |
|--------|------------|------------------|
| **Filter vs `ApiExceptionHandler`** | 2.4 | Problem bodies must serialize correctly in filters; document when MVC advice does not apply. |
| **401 vs 404 vs permitAll** | 2.6 | FR27 tests must respect `SecurityConfig` — unmapped paths under `permitAll` vs authenticated tree behave differently. |
| **JWT role lag** | 2.5 | After role change, access token may lag; UI uses server response as source of truth for rows — document for users. |
| **OpenAPI automated smoke** | 2.2–2.3 (deferred) | Optional manual verification accepted; consider one CI job when Epic 3 stabilizes API churn. |
| **Per-request DB lookup for account status** | 2.4 (deferred) | MVP tradeoff; revisit if latency or load becomes visible. |
| **Package layout vs architecture** | 2.2 | Flat `controller`/`service` vs `user/` package — team stayed consistent; migrate later if desired. |

### Deep story synthesis — recurring technical themes

- **Integration-first testing:** Every epic story emphasized HTTP + Problem Details `code` + tenant/role negatives in `*IT` classes.
- **Cross-story refinement:** 2.2 introduced `EmailNormalization`; 2.3 added pessimistic patterns for last-admin; 2.4 aligned **ACTIVE** admin counting with deactivation and PATCH role; 2.5–2.6 tightened docs and guardrail tests.
- **Technical debt logged in stories:** Deferred items (OpenAPI smoke, mobile E2E, large-org pessimistic lock) are explicit — not hidden.
