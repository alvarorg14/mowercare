## Part 1 — Epic review

### What went well

1. **Specification discipline:** Story files consistently map scope to `architecture.md`, Problem Details, OpenAPI, and UX-DR IDs — reduces rework between API and mobile.
2. **Clear sequencing:** Data model → bootstrap → auth endpoints → global JWT/tenant enforcement → mobile shell → real login → org profile demonstrates a coherent dependency chain.
3. **Tenant isolation early:** Story 1.5’s explicit cross-tenant denial tests support NFR-S3 before feature work piles on.
4. **Deferred work log:** `deferred-work.md` captures code-review follow-ups (pom metadata, Testcontainers lifecycle, DRY for `requestInstance`, fetch timeouts, etc.) without blocking merges — a reusable pattern.

### Challenges and patterns (from story Dev Notes + deferred items)

| Theme | Occurrence | Systems takeaway |
|--------|------------|------------------|
| **Intentional interim security** | Bootstrap / early API | Permit-all phases were scoped and later superseded by 1.5 — document when “temporary” security is product-approved. |
| **Edge cases deferred with rationale** | 1.4 refresh concurrency, hash collision handling | Acceptable for MVP when explicitly logged; re-visit if incident data says otherwise. |
| **Client resilience** | Mobile `fetch` / error swallowing for UX | Track under a future “HTTP policy” (timeouts, logging) so observability does not lag features. |
| **Consistency & DRY** | Duplicate helpers for Problem `instance` URI | Small refactors when touching error handling next — avoids silent drift. |
| **Operational polish** | Initializr placeholders in `pom.xml` | Fill when publishing or open-sourcing — not blocking product work. |
