## Part 1 — Epic review

### What went well

1. **Layered vertical slices:** 4.1 persistence only → 4.2 fan-out in-transaction → 4.3 first public APIs + tabs → 4.4 FCM + device tokens with no-op when Firebase missing → 4.5 navigation module + tests. Each story had a clear “no HTTP / no push” boundary in the spec.
2. **Explicit MVP matrix (4.2):** The locked routing table in the story file removed ambiguity about admins vs assignee vs actor exclusion and dedupe—reviews extended IT coverage instead of renegotiating product.
3. **Payload contract forward-looking (4.4):** FCM `data` keys (`organizationId`, `issueId`, …) were chosen so 4.5 did not require a backend redesign.
4. **Review culture carried forward:** Mark-read error handling, wrong-org PATCH tests, taxonomy label fallback, extended push-navigation branches—same pattern as Epic 3’s Review Findings checklists.
5. **Operational honesty:** Push failure does not roll back issue saves; invalid FCM tokens get removed; dev without credentials gets a clear log path—matches NFR-R2 intent.

### Challenges and patterns (from Dev Notes + Review Findings + deferrals)

| Theme | Where it showed up | Systems takeaway |
|--------|-------------------|-------------------|
| **Transaction + push latency** | 4.4 defer | Dispatch runs before commit; acceptable for MVP—watch for timeouts if send path grows heavier; after-commit or async queue is the natural evolution. |
| **Notification list pagination** | 4.3 defer | First page only (50 rows)—same class of debt as issue history “first 50” in Epic 3; product should own when “load more” matters. |
| **EAS / native push hardening** | 4.4 defer | Config plugins in repo; real store builds still need entitlement/channel verification on first release train. |
| **Fan-out rollback IT** | 4.2 defer | Atomicity is by design; failure-injected IT still a gap if you want provable rollback stories. |
| **`event_type` DB CHECK** | 4.1 defer | Application enforces taxonomy; optional DB constraint if you want stronger invariants at rest. |
| **Cold start / dedupe** | 4.5 implementation | OS can replay last response; one-shot guard is necessary—documented and tested in `push-navigation.test.ts`. |

### Deep story synthesis — recurring technical themes

- **Tenant + employee guards** mirrored inbox and device-token routes (`TenantPathAuthorization`, `RoleAuthorization`, JWT `sub` parsing hardening in 4.4 batch).
- **Pure rule logic** (fan-out matrix) separated for unit tests; persistence stays in Spring services—same testability pattern as earlier epics.
- **Mobile:** TanStack Query keys for org-scoped notification list; push path isolated in `push-navigation.ts` beside `notifications.ts`—keeps navigation concerns testable without spinning native modules in Jest.

## Key takeaways

1. **Transactional event + fan-out first** kept awareness consistent with the issue aggregate as source of truth and avoided orphan recipient rows.
2. **Stable FCM data contract** let 4.5 ship as a mobile-only navigation story—strong dependency ordering.
